# VEYA — Conversion Optimization

**כל מסך במערכת נבחן דרך חמישה עדשות.** לפני שמשחררים מסך/פיצ'ר, עבור עליהן.

## חמש העדשות
1. **Conversion** — האם המסך מוביל לפעולה הרצויה? יש CTA יחיד וברור?
   האם משהו מסיח או חוסם?
2. **Activation** — האם המשתמש מגיע לרגע ה"אהה" (הערך הראשון)? במהירות?
3. **Retention** — האם יש סיבה לחזור? המסך מזמין את הצעד הבא?
4. **Trust** — האם המסך משדר אמינות ובטיחות? (קריטי כפליים במתנות באשראי —
   נתוני מוזמנים ומידע פיננסי.)
5. **Revenue** — האם המסך תומך במסלול להכנסה (שדרוג/תשלום), בלי לפגוע בחוויה?

## רגעי ליבה למדוד ולשפר (Funnel)
נחיתה → הרשמה → Onboarding מושלם → **Activation** (ייבוא מוזמנים / שליחת הזמנה
ראשונה) → שימוש חוזר (RSVP, הושבה) → **המרה לתשלום** → מתנות/תודות. המדדים
המדויקים ב-`analytics-kpis.md`.

## עקרונות
- **CTA יחיד למסך.** ריבוי כפתורים שווי-משקל = פגיעה בהמרה.
- **הפחתת חיכוך** בכל שלב — כל שדה/קליק מיותר מוריד המרה.
- **Trust לפני בקשה** — הראה ערך/ביטחון לפני שאתה מבקש פרטים או תשלום.
- **רגע ה-Activation קדוש** — הכל מכוון להביא את המשתמש לערך ראשון מהר.
- **מדוד לפני שמשפרים** — ראה `analytics-kpis.md`; אין אינסטרומנטציה? זו
  שאלה פתוחה (`open-questions.md`).

## איך להשתמש בפועל
כשעובדים על מסך: כתוב שורה קצרה לכל עדשה ("Conversion: CTA 'שלחו הזמנות'
בולט; Activation: מגיעים לכאן ישר מ-Onboarding; Trust: מוקאפ הזמנה מרגיע").
אם עדשה נכשלת — שפר לפני שחרור. החלטות משמעותיות → `decisions.md`.
