# VEYA · תוכנית הפעלת RLS בייצור

**מצב: מסמך זה הוא תוכנית בלבד. שום פעולה כאן עדיין לא בוצעה על הייצור.
כל שלב דורש את האישור המפורש שלך לפני ביצוע — זה לא סקריפט להרצה
אוטומטית.**

## למה אנחנו כאן

כל השלבים הקודמים הושלמו בהצלחה:
- ✅ שלבים 1-3 (תיקוני RLS, מודל הרשאות, אימות תאימות אפליקציה) — RLS_REPORT.md
- ✅ מודל הרשאות מדויק, מיושר בין קוד ל-SQL — PERMISSIONS.md
- ✅ **בדיקה מלאה על Supabase staging אמיתי: 44/44 בדיקות עברו, 0 כשלים,
  יציב על 2 הרצות רצופות** — STAGING_TEST_REPORT.md. הבדיקה הזו גם גילתה
  ותיקנה 4 באגים אמיתיים (ראו שם) שלא ניתן היה לגלות בלי Postgres אמיתי.

זה השלב האחרון: להעביר את אותם השינויים בדיוק (קוד + SQL) לסביבת הייצור,
בזהירות ובשלבים, עם אפשרות חזרה מיידית בכל נקודה.

---

## עקרון מנחה: כל שלב הפיך בנפרד, לפני השלב הבא

```
שלב א: פריסת קוד (Render)         ← הפיך: git revert + deploy מחדש
שלב ב: הרצת SQL בייצור             ← הפיך: לא משנה התנהגות (postgres superuser עדיין עוקף RLS)
שלב ג: החלפת DATABASE_URL          ← זה הרגע שRLS "נדלק" בפועל
שלב ד: אימות + מעקב                ← אם יש בעיה: חזרה מיידית ל-DATABASE_URL הקודם
```

**השלב היחיד שבאמת "מדליק" את RLS הוא שלב ג'.** שלבים א'-ב' אינם
production-impacting כלל — אפשר לבצע אותם בלי סיכון, ולעצור שם אם תרצה
זמן נוסף לבדוק לפני שלב ג'.

---

## שלב א' — פריסת הקוד המתוקן לייצור

כל הקוד שתוקן בתהליך הזה (SECURITY DEFINER wrappers, `expire_on_commit`,
מודל ההרשאות, `app_create_event` וכו') כבר בענף העבודה שלך. יש לוודא
שהוא נפרס ל-Render **לפני** הרצת ה-SQL (השרת חייב להכיר את הפונקציות
שה-SQL יוצר).

**חשוב:** הקוד הזה **תואם-לאחור לחלוטין** — כל עוד `DATABASE_URL` עדיין
מצביע על `postgres` (המצב היום), שום התנהגות לא משתנה. אפשר לפרוס את
זה עכשיו, בלי שום סיכון, ולתת לו לרוץ בייצור כמה שרוצים לפני שלב ב'.

```bash
git push origin main   # או התהליך הרגיל שלך לפריסה ל-Render
```

לוודא ב-Render שהשרת עלה בהצלחה ושכל בדיקות התקינות (health check) עוברות.

## שלב ב' — הרצת קובצי ה-RLS על מסד הייצור

**עדיין לא production-impacting** — כל עוד `DATABASE_URL` הוא `postgres`
(superuser), RLS+FORCE לא משפיעים על שום דבר בפועל.

1. גבו את מסד הייצור באופן ידני (Supabase Dashboard → Database → Backups
   → Create backup now), **גם אם** ההרצה הבאה לא הרסנית (ליתר ביטחון).
2. ב-Supabase SQL Editor של פרויקט **הייצור** (לא ה-staging!):
   - פתחו את `backend/rls/01_helpers_and_grants.sql`.
   - **החליפו את `CHANGE_ME_STRONG_PASSWORD` בסיסמה חזקה ואקראית חדשה**
     — **שונה** מזו ששימשה ב-staging. שמרו אותה במקום מאובטח (מנהל
     סיסמאות) — היא תיכנס בשלב ג' ל-`DATABASE_URL` של הייצור.
   - הריצו את הקובץ במלואו.
   - הריצו את `backend/rls/02_policies.sql` במלואו.
3. בדיקת עשן (עדיין כ-postgres, אמור להראות RLS מופעל על כל הטבלאות):
   ```sql
   select relname from pg_class
   where relnamespace='public'::regnamespace and relkind='r' and relrowsecurity
   order by 1;
   ```
   צריך להחזיר 10 טבלאות: `events, guests, messages, clarifications,
   automation_rules, message_templates, audit_logs, event_members, users,
   login_events`.
4. **עצרו כאן.** האפליקציה עדיין מחוברת כ-`postgres` (superuser) — שום
   דבר לא השתנה עבור המשתמשים האמיתיים. זה זמן טוב לעצור, לישון על זה,
   ולחזור לשלב ג' רק כשנוח לך (עדיף בשעה עם תנועה נמוכה, למקרה הצורך
   ב-rollback).

## שלב ג' — המעבר בפועל: החלפת `DATABASE_URL` לתפקיד `veya_app`

**זה הרגע ש-RLS מתחיל להשפיע על משתמשים אמיתיים.** מומלץ לבצע בשעת
תנועה נמוכה, עם מישהו זמין לעקוב אחרי הלוגים ב-15 הדקות הראשונות.

ב-Render → השרת של הייצור → Environment:
1. הוסיפו משתנה סביבה חדש: `MIGRATIONS_DATABASE_URL` = אותו connection
   string שהיה קודם ב-`DATABASE_URL` (כלומר, חיבור **postgres/superuser**
   — הוא ימשיך לשמש למיגרציות/זריעה בעליית השרת).
2. שנו את `DATABASE_URL` עצמו לחיבור **`veya_app`** עם הסיסמה שיצרתם
   בשלב ב': `postgresql://veya_app:[הסיסמה]@[host]:5432/postgres` (או
   פורמט ה-pooler הרלוונטי לפרויקט הייצור — ראו הערה למטה).
3. Deploy / Restart את השרת כדי שהמשתנים החדשים ייטענו.

**הערה על פורמט חיבור ה-pooler:** בבדיקת ה-staging גילינו שחיבור דרך
ה-pooler של Supabase דורש שם משתמש בפורמט `<role>.<project-ref>`
(למשל `veya_app.xxxxxxxx`), לא רק `veya_app`. ודאו זאת מול הפרויקט הספציפי
של הייצור ב-Supabase Dashboard → Settings → Database → Connection Pooling,
לפני שממלאים את הערך.

## שלב ד' — אימות מיידי (5-15 הדקות הראשונות)

מיד אחרי ה-deploy, בצעו (או בקשו ממישהו לבצע) ידנית:
1. **הרשמה** — משתמש/ת בדיקה חדש/ה נרשם/ת בהצלחה (לא 500).
2. **כניסה** — משתמש קיים מתחבר בהצלחה.
3. **דשבורד** — בעל אירוע קיים רואה את הנתונים שלו (מוזמנים, סטטיסטיקות).
4. **קישור RSVP ציבורי** — פותחים קישור `/confirm/{token}` אמיתי (של
   מוזמן קיים) ומוודאים שהוא עדיין עובד ומאפשר אישור הגעה.
5. **לוגים** — עוקבים אחרי לוגי Render לחיפוש `row-level security policy`
   או שגיאות 500 חדשות שלא היו לפני כן.

אם **הכול תקין** → שלב ה' (מעקב מורחב). אם **משהו נשבר** → Rollback מיידי
(ראו למטה) ואז חקירה על עותק/staging, לא על הייצור החי.

## שלב ה' — מעקב מורחב (24-48 שעות)

- עקבו אחרי שיעור השגיאות (500) הכללי של ה-API בהשוואה לפני ההפעלה.
- בדקו במיוחד: שליחת הזמנות WhatsApp אמיתיות (אם כבר לא במצב mock),
  ו-webhook תשובות RSVP נכנסות — אלה התרחישים המורכבים ביותר שנבדקו.
- אם יש חברי-אירוע (מפיקים/אולמות) פעילים בייצור עם הרשאות חלקיות —
  זו הפעם הראשונה שהם ייבדקו תחת RLS אמיתי במלואם. עקבו במיוחד אחריהם.

## תוכנית Rollback (בכל שלב אחרי ג')

**מיידי (שניות, בלי לגעת ב-DB):** ב-Render, מחזירים את `DATABASE_URL`
לחיבור `postgres` הישן (superuser) ועושים deploy מחדש. superuser עוקף
RLS לגמרי — ההתנהגות חוזרת מיידית למה שהייתה לפני שלב ג'. **זו דרך
ה-rollback הראשית והמומלצת** — לא צריך להריץ שום SQL נוסף.

**מלא (אופציונלי, רק אם רוצים לכבות גם את מנגנון ה-RLS ב-DB עצמו):**
הריצו את `backend/rls/03_rollback.sql` על מסד הייצור.

## מה לעשות אם מתגלה בעיה שלא נראתה ב-staging

זה קורה — staging מדמה אך לא משוכפל 1:1 את הייצור (למשל: נפח נתונים,
דפוסי שימוש אמיתיים). אם מתגלה בעיה:
1. Rollback מיידי (ראו למעלה) — **קודם** מחזירים שירות, **אחר כך** חוקרים.
2. שחזרו את הבעיה על סביבת ה-staging (שנשארת קיימת ונקייה, ראו
   STAGING_PLAN.md) — לא על הייצור החי.
3. תקנו, בדקו מול staging, ורק אז חזרו לשלב ג' עם התיקון.

## סיכום — מה עדיין דורש את האישור שלך

| פעולה | האם production-impacting | דורש אישור מפורש |
|---|---|---|
| פריסת הקוד המתוקן (שלב א') | לא | כן, לפני כל push לייצור |
| הרצת SQL על מסד הייצור (שלב ב') | לא (superuser עדיין עוקף RLS) | כן |
| החלפת `DATABASE_URL` ל-`veya_app` (שלב ג') | **כן — זה הרגע הקריטי** | **כן, במפורש, בנפרד משלבים א'-ב'** |
| Rollback (בכל שלב) | תלוי בכיוון | כן, גם rollback הוא פעולת ייצור |

**עד כה לא בוצעה אף פעולה על מסד הייצור. הכול ממתין לאישורך, שלב אחר שלב.**
